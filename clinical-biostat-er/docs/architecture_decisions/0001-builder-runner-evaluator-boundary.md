# ADR 0001: Builder-Runner-Evaluator Boundary

## Status

Accepted.

## Context

The mock01 review work exposed an unstable pattern: deliverable plotting logic
could be generated by Claude Code during a run. That is not a stable skill
ecosystem. Repeated clinical/statistical plotting logic must live in the
builder-owned skill/runtime library, with explicit contracts and evaluator
checks.

## Decision

The bundle uses three roles:

- Builder: Xihao + Codex maintain skills, helper libraries, manifests,
  contracts, review gates, and evaluator harnesses.
- Runner: Claude Code executes prepared tasks by selecting skills and calling
  builder-owned helpers. It must not invent deliverable plotting or statistical
  implementations at run time.
- Evaluator: builder-owned R validators, comparison scripts, and acceptance
  runners verify runner behavior and generated artifacts. Claude Code is the system under test, not the judge of its own output.

Deliverable figure generation must be routed through
`docs/review_evidence/plot_capability_ownership_map.csv`.

The current mock01/Core2 figure boundary covers nine plot classes and 54
figures:

- Core2 `individual_profile` and `swimmer_event_overlay` call direct AZ Rmd
  extracts `core2_az_create_individual_pk_plot()` /
  `core2_az_create_swimmer_plot()` through
  `core2_render_reference_figure_previews()`; their plotting code is direct
  extract, while their dat_* input adapter remains review-gated.
- Core4 `er_pair_three_panel` calls direct AZ Rmd extract
  `core4_az_create_combined_er_plot()` /
  `core4_export_mock01_er_pair_figures_from_root()`.
- Core5 KM/Cox/TTE plot classes call direct AZ Rmd extract
  `core5_az_export_mock01_km_cox_figures()` /
  `core5_export_mock01_km_cox_figures_from_root()`.

Every plot capability row must declare `runner_may_inline_code = no`.

## Enforcement

- `run_prepared_claude_case.R` writes `runner_inline_plotting_audit.csv` during
  live prepared-case execution and records
  `runner_inline_plotting_code_detected` when run-local R/Rmd scripts look like
  new deliverable plotting implementations.
- `prepare_claude_case_run.R` appends a prepared-case plot capability boundary
  to every generated `prompt.md` and `RUNBOOK.md`, and records the ownership map
  path in `case_run_manifest.csv`.
- `run_mock01_review_acceptance.R` runs the ownership-map builder and requires
  nine plot classes, 54 figures, non-empty helper/evaluator guards, and
  `runner_may_inline_code = no` for every row.
- `build_review_packet.R` includes the ownership map and README in the review
  packet.

## Consequences

Claude Code runner failures caused by missing plot capabilities should be fixed
by the builder adding or improving the skill/runtime helper, not by asking the
runner to improvise a local plotting implementation.

This decision does not claim that all figures have final visual parity. Core4
and Core5 semantic ports still require plotted-data/layer parity work, and
Core2 profile/swimmer previews remain clinical-semantics review-gated.
